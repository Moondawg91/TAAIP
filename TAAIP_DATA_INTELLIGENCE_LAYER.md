# TAAIP Data Intelligence Layer Implementation

## Overview

The TAAIP Data Intelligence Layer is a production-ready operational intelligence system that unifies fragmented RID/Vantage/PowerBI/EMM data into a single decision-support ecosystem for USAREC/USAR recruiting operations.

**Core Doctrine:**
- Preserves historical snapshots - no destructive overwrites
- Maintains relational linkage across all modules
- Supports commander decision-making with advisory (non-autonomous) recommendations
- Enables recruiter operational effectiveness analytics
- Supports market intelligence overlays
- Tracks targeting → operations → field activities → funnel → ROI → performance → reassessment

## Architecture

### 1. Data Ingestion & Normalization

**Module:** `intelligence_ingestion.py`

Handles ingestion from multiple data sources:
- RID (Recruiting Information Database) - CSV/XLSX
- Vantage/Foundry exports - CSV/XLSX
- PowerBI exports - CSV/XLSX
- EMM exports - CSV/XLSX
- Custom imports

**Features:**
- Auto-detects schema from source files
- Maps source columns to normalized fields
- Normalizes categorical values (ZIPs, RSIDs, schools, companies)
- Preserves raw source files for audit trail
- Creates historical lineage record

**Key Classes:**
- `SchemaDetector` - Auto-detects column types and suggests normalized field names
- `ValueNormalizer` - Normalizes categorical values with confidence scores
- `IngestionPipeline` - Main ingestion orchestrator

**Process Flow:**
```
CSV/XLSX Upload → SchemaDetector → ColumnMapping → ValueNormalizer → DataIngestionLog
                                  ↓
                         NormalizedValue records
```

### 2. Historical Snapshots

**Module:** `intelligence_analytics.py` (HistoricalSnapshotEngine)

Creates daily/weekly/monthly snapshots that are never overwritten.

**Features:**
- Daily, weekly, monthly snapshots
- Event-triggered snapshots (operation start/end, board decisions)
- Multiple snapshots per date (tracked by version number)
- Stores arbitrary KPIs as snapshot metrics
- Enables trend analysis and predictive modeling

**Database Tables:**
- `historical_snapshots` - Snapshot metadata and versioning
- `snapshot_metrics` - Individual metric values within snapshots

### 3. Out-of-Area Contract Analytics

**Module:** `intelligence_analytics.py` (OutOfAreaContractAnalytics)

Classifies and analyzes contracts to understand market dynamics, leakage, and operational influence.

**Classifications:**
- `in_area` - Contract applicant and assignment ZIP match writing RSID territory
- `out_of_area` - Contract assigned outside writing RSID territory
- `imported` - Contract generated by external influence
- `exported` - Contract represents market penetration into other territories
- `cross_market` - Cross-RSID contract with multi-market influence

**Analyses:**
1. **Contract Classification** - Assign each contract to a classification category
2. **Market Leakage** - Track ZIP-to-ZIP contracts lost from territory
3. **Contract Influence** - Identify which RSIDs/ZIPs influence others' success

**Metrics Calculated:**
- Market penetration score (0.0-1.0)
- Territory control score (0.0-1.0)
- Operational influence score (0.0-1.0)
- Leakage tracking by leak type
- Influence scoring with causation confidence

**Database Tables:**
- `contract_classifications` - Individual contract classifications
- `market_leakage` - ZIP-to-ZIP leakage tracking
- `contract_influence` - Cross-market influence analysis

### 4. Recruiter Operational Effectiveness

**Module:** `intelligence_analytics.py` (RecruiterEffectivenessAnalytics, PredictiveProductionPacingEngine)

Calculates objective operational metrics for recruiter performance analysis.

**Key Principle:** All metrics are objective and non-judgmental. No subjective language (e.g., "lazy").

**Metrics Calculated:**
- Prospecting hours (input from activity records)
- Contacts per hour
- Appointments per hour
- Contracts per hour
- Attempts per hour
- Hours per appointment
- Hours per enlistment
- Contact conversion rate (contacts → appointments)
- Appointment conversion rate (appointments → contracts)
- Overall conversion rate (prospecting → contracts)
- Workload % of station
- Efficiency index (0.0-1.0, benchmarked to peers)
- Effort index (0.0-1.0, hours worked vs peers)

**Predictive Production Pacing:**
- Forecasts recruiter production for weekly/monthly/quarterly periods
- Uses 30/90/180-day lookback for trend analysis
- Includes confidence levels based on data recency
- Calculates pacing vs goal gaps
- Provides actionable production forecasts

**Database Tables:**
- `recruiter_activity` - Raw recruiter activities (contacts, appointments, etc)
- `recruiter_effectiveness` - Calculated effectiveness metrics
- `predictive_production_pace` - Production forecasts with confidence

### 5. Vacancy Alignment Engine

**Module:** `intelligence_recommendations.py` (VacancyAlignmentEngine)

Aligns military vacancies to market opportunities.

**Alignment Analysis:**
- Demographic fit score
- School population fit score
- Civilian industry alignment score
- Operational trend alignment score
- Overall alignment score (average of above)

**Creates:**
- Target population recommendations
- Messaging theme recommendations
- Marketing platform recommendations
- Event type recommendations
- Partnership opportunities

**Output:** Structured vacancy alignment records with actionable recommendations.

### 6. ROP/SRP Recommendation Engine

**Module:** `intelligence_recommendations.py` (RopSrpRecommendationEngine)

Generates advisory recommendations for commanders.

**Recommendation Types:**
1. **School Effectiveness** - ROP/SRP changes based on school production trends
2. **Recruiter Realignment** - Coaching/task rebalancing based on effectiveness
3. **Market Leakage** - Actions to recover leaked market share
4. **Operation Adjustment** - General operational improvements

**Recommendation Structure:**

Each recommendation includes:
- **Rationale** - Explains WHY, WHAT DATA supports it, HOW it impacts operations
  - Plain English summary of the recommendation
  - Supporting data points with thresholds and trends
  - Impact analysis (financial, operational)
  - Confidence score (0.0-1.0)

- **Advisory Details**
  - Recommendation type and scope
  - Priority (critical, high, medium, low)
  - Urgency (immediate, 30_days, 90_days, ongoing)
  - Detailed parameters/configuration

- **Commander Decision**
  - Status tracking (advisory → acknowledged → implemented/rejected/deferred)
  - Decision capture (accepted/modified/rejected)
  - Implementation tracking (start/end dates)

**Critical Design:** All recommendations are ADVISORY ONLY. Commanders retain full authority over decisions.

**Database Tables:**
- `recommendation_rationale` - Structured rationale for each recommendation
- `advisory_recommendations` - Recommendation records with decision tracking

## API Endpoints

All endpoints are located at `/api/v2/intelligence/` and are non-breaking additions to existing APIs.

### Ingestion Endpoints

```
POST /api/v2/intelligence/ingest/file
  - source_name: RID, Vantage, PowerBI, EMM, or custom
  - file: CSV/XLSX file upload
  - ingested_by: Optional user/system identifier
  - metadata: Optional source metadata
  → Returns: ingestion log with record count and status

GET /api/v2/intelligence/ingestion-logs
  - source_name: Optional filter by source
  - limit: Result limit (default 50)
  → Returns: Historical ingestion records for audit trail
```

### Historical Snapshot Endpoints

```
POST /api/v2/intelligence/snapshots
  - snapshot_type: daily, weekly, monthly, event_triggered
  - snapshot_date: Date of snapshot
  - scope_type: USAREC, BRIGADE, BATTALION, COMPANY, STATION
  - scope_value: Scope identifier (brigade_prefix, station_rsid, etc)
  - trigger_event: Optional trigger (operation_start, operation_end, etc)
  - metrics: Optional dict of KPI name → value
  → Returns: Created snapshot record

GET /api/v2/intelligence/snapshots
  - scope_type, scope_value, start_date, end_date: Filters
  - limit: Result limit (default 100)
  → Returns: Snapshot records (ordered by date DESC)
```

### Out-of-Area Contract Analytics

```
POST /api/v2/intelligence/contracts/classify
  - contract_id, applicant_zip, assigned_zip, writing_rsid, assigned_rsid
  - school_zip, event_zip, originating_operation
  → Returns: Contract classification with scores and confidence

GET /api/v2/intelligence/market-leakage
  - from_zip, from_rsid: Filters
  - start_date, end_date: Period filters
  - limit: Result limit (default 50)
  → Returns: Market leakage records (ordered by date DESC)
```

### Recruiter Effectiveness

```
POST /api/v2/intelligence/recruiter-activity
  - recruiter_id, activity_date, activity_type
  - activity_count, activity_duration_hours, outcome_count
  - station_rsid, source_system
  → Returns: Activity record created

POST /api/v2/intelligence/recruiter-effectiveness
  - recruiter_id, period_date, reporting_period (daily/weekly/monthly)
  - station_rsid: Optional
  → Returns: Effectiveness metrics (contacts/hr, contracts/hr, efficiency, etc)

GET /api/v2/intelligence/recruiter-effectiveness/{recruiter_id}
  - start_date, end_date: Period filters
  - limit: Result limit (default 12)
  → Returns: Historical effectiveness records

POST /api/v2/intelligence/recruiter-production-forecast
  - recruiter_id, forecast_period (weekly/monthly/quarterly)
  - station_rsid: Optional
  → Returns: Production forecast with confidence level and pacing gap
```

### Vacancy Alignment & Recommendations

```
POST /api/v2/intelligence/vacancy-alignment
  - vacancy_mos, vacancy_count, market_zip_primary
  - station_rsid: Optional
  → Returns: Alignment analysis with fit scores and recommendations

GET /api/v2/intelligence/recommendations
  - recommendation_type, status, scope_type, priority: Filters
  - limit: Result limit (default 50)
  → Returns: Advisory recommendations (filtered and ordered)

POST /api/v2/intelligence/recommendations/{recommendation_id}/acknowledge
  - commander_id
  → Returns: Recommendation acknowledgment record

POST /api/v2/intelligence/recommendations/{recommendation_id}/decision
  - decision (accepted/modified/rejected)
  - decision_notes, commander_id: Optional
  → Returns: Recommendation decision record
```

## Operational Constraints & Design Principles

1. **No Destructive Overwrites**
   - All data is additive only
   - Snapshots are versioned and preserved
   - Normalization mappings are auditablе

2. **Non-Autonomous Decisions**
   - All recommendations are ADVISORY
   - Commanders retain full authority
   - Decision audit trail is maintained

3. **Objective Analysis**
   - No subjective language in metrics
   - All analysis grounded in data
   - Confidence scores on all recommendations

4. **Preserved Lineage**
   - Raw source files preserved
   - Ingestion metadata recorded
   - Classification confidence tracked
   - Rationale documented for recommendations

5. **Non-Breaking API**
   - All endpoints under `/api/v2/intelligence/`
   - No changes to existing API signatures
   - No backend call modifications
   - No routing changes

## Integration Points

The Data Intelligence Layer integrates with existing TAAIP systems:

1. **Events** - Classification and ROI analysis input
2. **Funnel Transitions** - Contract data for leakage analysis
3. **Recruiter Activity** - Effectiveness calculation input
4. **Marketing Activities** - Vacancy alignment input
5. **Burden/Mission Data** - Operational context

All integrations are read-only from existing systems. No modifications to existing entity structures or relationships.

## Performance Considerations

1. **Ingestion** - Processes files in batches; preserves raw data separately
2. **Analytics** - Pre-calculates metrics for quick querying
3. **Snapshots** - Time-series queries optimized for trend analysis
4. **Recommendations** - Generated periodically or on-demand; cached

## Security & Authorization

- All endpoints require existing TAAIP authentication
- Authorization based on existing role-based access control
- Audit trail maintained for all ingestions and decisions
- No sensitive data stored in recommendations; only references
